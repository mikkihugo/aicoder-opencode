### M58: `ModelRegistryPlugin` factory never called `initializeProviderHealthState` at startup — every auth.json + env-var credential check was dead code at runtime, and `key_missing` entries with `until: Infinity` crashed `new Date(until).toISOString()` at 7 render sites the moment any were installed `✅ COMPLETED`

Completion Notes (2026-04-11):
- **Bug observed**: `ModelRegistryPlugin` at `src/plugins/model-registry.ts` line 2410 called `loadPersistedProviderHealth()` directly and never invoked `initializeProviderHealthState`. `initializeProviderHealthState` was defined and exported at line 2408 (`export { inferTaskComplexity, recommendTaskModelRoute, initializeProviderHealthState };`) — but ONLY for the test suite to import. The production factory never touched it. The downstream effect is that `loadAuthKeys()`, `providerHasEnvVarCredential()`, and the M33 stale-`key_missing` reconciliation loop — all of the M12/M13/M33/M55 auth.json/env-var detection machinery — never ran at runtime. On plugin startup with a fresh install or a freshly-cleared health file, the `providerHealthMap` came back completely empty. The fallback scanner (`findFirstHealthyVisibleRoute` from M57, `findPreferredHealthyRoute` from M56) has no way to know a provider has no credentials, so it happily routed traffic through uncredentialed providers and burned retries on 401/403 responses before the runtime feedback path could mark them dead. All the tests that pinned `initializeProviderHealthState`'s correctness passed because they imported and called it DIRECTLY — not the factory. Nothing tested the factory's bootstrap sequence, so the wire-up rotted silently for months. Second-order bug exposed by the fix: seven render sites (`computeRegistryEntryHealthReport` × 2, `buildProviderHealthSystemPrompt` × 2, `get_quota_backoff_status` × 2, `recommend_model_for_role` × 1) all called `new Date(health.until).toISOString()` without guarding against `Number.POSITIVE_INFINITY` — the sentinel `key_missing` uses for "no expiry." - **Reachability**: 100% reachable on every plugin cold-start. The failure mode is "fallback scanner routes around live penalty states but happily picks uncredentialed providers." Classic dead-wiring bug where the unit tests pass and the integration path is broken. Observed in practice after the openrouter-key-renewal incident (`openrouter` key dead, plugin never marked it `key_missing`, kept handing out `openrouter/*:free` routes that 401'd).
- **Fix**: Two coupled changes that must ship together.
  1. Extracted a new exported helper `initializeRuntimeProviderState(loadRegistry, logError)` that composes `loadRegistry()` with `initializeProviderHealthState(modelRegistry.models)`, falls back to `initializeProviderHealthState([])` + `logError(error)` when the registry load throws (so the plugin always boots). Rewired `ModelRegistryPlugin` to call `initializeRuntimeProviderState(() => loadModelRegistry(CONTROL_PLANE_ROOT_DIRECTORY))` instead of `loadPersistedProviderHealth()`. The helper takes the loader as a DI parameter so tests can supply a fixed registry or throwing loader without constructing a disk fixture.
  2. Extracted a new exported helper `formatHealthExpiry(until)` that returns `"never"` for `!Number.isFinite(until)` and `new Date(until).toISOString()` otherwise. Replaced all 7 sites that crashed on `Infinity`.
- **Tests**: four new tests in `src/plugins/model-registry.keyless.test.ts`.

This is the test that catches BOTH coupled bugs — the wiring gap AND the Infinity-render crash — through the real production call path.

### M57: `recommendTaskModelRoute` last-resort fallback scanned only `roleMatchedEntries` (already narrowed by `allowedTiers` from `complexity`), so a request whose complexity tier was fully throttled threw "No healthy model route found" even when a sibling-tier entry with the same role was perfectly healthy — graceful degradation was impossible and the caller saw a hard failure when a working alternative was one tier removed `✅ COMPLETED`

Completion Notes (2026-04-11):
- **Bug observed**: `recommendTaskModelRoute` in `src/plugins/model-registry.ts` (pre-M57, lines 2194-2205) implemented the last-resort fallback as an inline loop over `roleMatchedEntries` — the SAME list used by `selectBestModelForRoleAndTask` above, already filtered through `enabled` + `role` + `allowedTiers[complexity]`. When every visible route inside that narrow tier set had a live penalty (provider `quota`/`key_dead`/`no_credit`/`key_missing`, or route-level `model_not_found`/`timeout`/`quota`), the loop completed without returning and control fell through to `throw new Error(`No healthy model route found for agent '${task.subagent_type}' with role '${role}' and complexity '${complexity}'`)`. The throw propagates out of the plugin hook and kills whatever called in — agent dispatch, routing-recommendation tool, or system-prompt injection — as a hard failure. A standard-tier architect entry sitting right there in `modelRegistryEntries` with a healthy `opencode/*-free` route was invisible to the fallback scan because `standard` wasn't in `allowedTiers` for a `large`-complexity request. No warning, no log line, no degradation path — just termination.
- **Reachability**: reachable today under any multi-provider outage. `allowedTiers` for `large` complexity is `["strong", "frontier"]`; for `small` it is `["tiny", "fast", "standard"]`; for `medium` it is `["standard", "strong"]`. A large-complexity request during a frontier+strong outage (all `ollama-cloud/*` routes timed out + `iflowcn` quota-throttled + `openrouter/*:free` all dead — the shape we've repeatedly seen during overlapping provider quota resets) has zero healthy entries in its tier set, and the scan throws even though `opencode/glm-4.7-free` (standard tier) is live and advertised for the `architect` role. The failure is the opposite of fail-safe: instead of "use a less-capable but working route," the caller sees "no model at all." Same audit shape as M48 / M52-M56: the narrow predicate fires silently in a real failure mode, the correct behavior is explicit graceful degradation.
- **Fix**: extracted a pure exported helper `findFirstHealthyVisibleRoute(candidateEntries, providerHealthMap, modelRouteHealthMap, now)` that returns the first `{provider, model}` whose provider AND composite route-health entry are both live, or `null` when none exist. Kept tier-agnostic by contract — the caller owns the candidate-set filter. `recommendTaskModelRoute` now runs the last-resort scan in two passes: first over `roleMatchedEntries` (preserves the existing within-tier preference — a large-complexity request continues to prefer strong/frontier routes whenever any are live), then over `rolePreferredEntries` (the `enabled + role` only set already computed for the M56 preferred-models lookup, no tier filter), with distinct reasoning strings so downstream telemetry can distinguish "within tier" from "outside tier" fallbacks. Only both passes returning `null` triggers the throw. The reasoning string `"Fallback to first healthy visible route outside complexity tier"` deliberately surfaces the degradation so operators see WHY a small-complexity prompt picked up a strong-tier route or a large-complexity prompt picked up a standard-tier route.
- **Tests**: six new tests in `src/plugins/model-registry.test.ts`:
  - `findFirstHealthyVisibleRoute_whenCandidateListEmpty_returnsNull` — empty-list baseline.
  - `findFirstHealthyVisibleRoute_whenFirstEntryHasHealthyRoute_returnsFirstMatch` — primary happy path (array order honored).
  - `findFirstHealthyVisibleRoute_whenFirstEntryProviderUnhealthy_returnsNextHealthyEntry` — provider-health skip pin.
  - `findFirstHealthyVisibleRoute_whenRouteLevelPenaltyBlocksPrimaryButSiblingHealthy_returnsSibling` — route-level-penalty skip pin (walks `provider_order` siblings inside one entry before moving to the next).
  - `findFirstHealthyVisibleRoute_whenEveryCandidateIsUnhealthy_returnsNull` — no-match fallthrough pin.
  - `recommendTaskModelRoute_whenComplexityTierFullyDownButSiblingTierHealthy_degradesToSiblingTier` — **headline integration pin**: end-to-end flow with a real temp-directory architect agent declaring `routing_complexity: large`, a registry where the frontier + strong entries have live penalties (ollama-cloud timeout route-health, iflowcn quota provider-health) and the standard-tier entry is fully healthy, and an expectation that the decision is `opencode/glm-4.7-free` with reasoning matching `/outside complexity tier/`. Pre-M57 this throws; post-M57 it degrades.
- **Verification**: verify-on-HEAD by reverting the tier-agnostic second pass (deleting the `tierAgnosticRoute = findFirstHealthyVisibleRoute(rolePreferredEntries, ...)` block so only the tier-filtered first pass runs before throw) and re-running `npx tsx --test src/plugins/model-registry.test.ts` — exactly **1 expected failure** fired: the headline integration pin. All five `findFirstHealthyVisibleRoute_*` unit pins still passed against the reverted caller — exactly as expected, because the helper itself is correct; only the caller's choice of whether to make a second pass matters for the bug. Restored the fix, 164/164 pass.
- **Files**: `src/plugins/model-registry.ts` (new exported `findFirstHealthyVisibleRoute` helper + docstring explaining the tier-agnostic contract, inline last-resort loop in `recommendTaskModelRoute` replaced with two-pass helper call that widens from `roleMatchedEntries` to `rolePreferredEntries`), `src/plugins/model-registry.test.ts` (6 new tests + added `findFirstHealthyVisibleRoute` to imports).

### M56: `recommendTaskModelRoute` preferred-models lookup iterated `roleMatchedEntries` (already narrowed by the complexity-tier filter), so an agent declaring a frontier-tier `model:` preference on a prompt that `inferTaskComplexity` classified as `"small"` silently had its preference dropped and the task was routed to a tiny/fast tier the agent never asked for `✅ COMPLETED`

Completion Notes (2026-04-11):
- **Bug observed**: `recommendTaskModelRoute` in `src/plugins/model-registry.ts` (pre-M56, lines 2044-2081) walked the agent's `preferredModels` list by iterating `roleMatchedEntries` — a list already filtered through BOTH the role filter AND the capability-tier filter from the inferred-or-explicit complexity. An agent that declared `model: minimax/MiniMax-M2.5` (a `frontier`-tier entry) with no explicit `routing_complexity:` in frontmatter, and received a prompt that `inferTaskComplexity` routed to `"small"` (e.g. "fix typo in README" — `typo` is a `SMALL_COMPLEXITY_KEYWORD_STEM`), saw its MiniMax-M2.5 entry excluded from `roleMatchedEntries` because `allowedTiers = ["tiny", "fast", "standard"]` did not contain `frontier`. The preferred-models loop found no match against the narrowed set, silently fell through to `selectBestModelForRoleAndTask`, and routed the task to whatever tiny/fast/standard alternative happened to be healthy — a model the agent author never asked for. The failure was invisible: no warning, no log line, the reasoning string in the final decision even claimed "Best registry model for role X and complexity Y" as if the preference had never been declared.
- **Reachability**: reachable today. Any agent under `.opencode/agents/` that (a) declares `model:` or `models:` pointing at a tier above `standard` and (b) omits `routing_complexity:`, is one matching keyword away from silently losing its preference. The M50 SMALL complexity branch widens the hit surface — it added `typo`, `rename`, `trivial`, `minor`, `whitespace` as stems, so prompts containing any of those words (a very common shape for maintenance tasks delegated to a specialist agent) now trigger the bug where they used to default to `medium`. The specific victims are the agents in the model catalog that legitimately want a frontier model for EVERY task they handle — e.g. a security_reviewer or critical_reviewer agent that declares a strong/frontier preference precisely so it doesn't miss anything. Same audit shape as M48, M52, M53, M54, M55: the narrow predicate's failure mode is silent and the correct behavior is explicit author intent over inferred heuristic.

### M16: Long successful reasoning turns silently marked `timeout` (ROOT CAUSE of strong-model starvation) `✅ COMPLETED`
